## Description

Adversaries may abuse built-in CLI tools or API calls to execute malicious commands in containerized environments.<br><br>The Docker CLI is used for managing containers via an exposed API point from the `dockerd` daemon. Some common examples of Docker CLI include Docker Desktop CLI and Docker Compose, but users are also able to use SDKs to interact with the API. For example, Docker SDK for Python can be used to run commands within a Python application.[^4] <br><br>Adversaries may leverage the Docker CLI, API, or SDK to pull or build Docker images (i.e., [[kb/mitre/attack/techniques/T1105-Ingress_Tool_Transfer\|Ingress Tool Transfer]], [[kb/mitre/attack/techniques/T1612-Build_Image_on_Host\|Build Image on Host]]), run containers (i.e., [[kb/mitre/attack/techniques/T1610-Deploy_Container\|Deploy Container]]), or execute commands inside running containers (i.e., [[kb/mitre/attack/techniques/T1609-Container_Administration_Command\|Container Administration Command]]). In some cases, threat actors may pull legitimate images that include scripts or tools that they can leverage - for example, using an image that includes the `curl` command to download payloads.[^1]  Adversaries may also utilize `docker inspect` and `docker ps` to scan for cloud environment variables and other running containers (i.e., [[kb/mitre/attack/techniques/T1613-Container_and_Resource_Discovery\|Container and Resource Discovery]]).[^3] [^2] <br><br>Kubernetes is responsible for the management and orchestration of containers across clusters. The Kubernetes control plane, which manages the state of the cluster and is responsible for scheduling, communication, and resource monitoring, can be invoked directly via the API or indirectly via CLI tools such as `kubectl`. It may also be accessed within client libraries such as Go or Python. By utilizing the API, administrators can interact with resources within the cluster such as listing or creating pods, which is a group of one or more containers. Adversaries call the API server via `curl` or other tools, allowing them to obtain further information about the environment such as pods, deployments, daemonsets, namespaces, or sysvars.[^2]  They may also run various commands regarding resource management.




## Mitigations
| ID | Name | Description |
| --- | --- | --- |
| [[kb/mitre/attack/mitigations/M1026-Privileged_Account_Management\|M1026]] | Privileged Account Management | Restrict permissions on API access. RBAC in Kubernetes involve permissions that are additive, meaning there are no explicit "deny" rules. These permissions can be defined within a particular namespace or within cluster-scoped resources. Securing the Docker daemon can be done by using SSH or TLS with certificate authorization. Container management tools such as Docker and Podman may offer ways to run containers as rootless, which prevents them from running with privileged permissions. |
| [[kb/mitre/attack/mitigations/M1038-Execution_Prevention\|M1038]] | Execution Prevention | Deny scripting where appropriate. Tools such as Python or Go can utilize Kubernetes and Docker within a client library and execute commands within their application. |
