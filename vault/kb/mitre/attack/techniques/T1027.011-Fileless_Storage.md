## Description

Adversaries may store data in "fileless" formats to conceal malicious activity from defenses. Fileless storage can be broadly defined as any format other than a file. Common examples of non-volatile fileless storage in Windows systems include the Windows Registry, event logs, or WMI repository.[^3] [^6]  Shared memory directories on Linux systems (`/dev/shm`, `/run/shm`, `/var/run`, and `/var/lock`) and volatile directories on Network Devices (`/tmp` and `/volatile`) may also be considered fileless storage, as files written to these directories are mapped directly to RAM and not stored on the disk.[^8] [^7] [^2] [^1] [^5] .<br><br>Similar to fileless in-memory behaviors such as [[kb/mitre/attack/techniques/T1620-Reflective_Code_Loading\|Reflective Code Loading]] and [[kb/mitre/attack/techniques/T1055-Process_Injection\|Process Injection]], fileless data storage may remain undetected by antivirus and other endpoint security tools that can only access specific file formats from disk storage. Leveraging fileless storage may also allow adversaries to bypass the protections offered by read-only file systems in Linux.[^4] <br><br>Adversaries may use fileless storage to conceal various types of stored data, including payloads/shellcode (potentially being used as part of [[kb/mitre/attack/tactics/TA0003-Persistence\|Persistence]]) and collected data not yet exfiltrated from the victim (e.g., [[kb/mitre/attack/techniques/T1074.001-Local_Data_Staging\|Local Data Staging]]). Adversaries also often encrypt, encode, splice, or otherwise obfuscate this fileless data when stored. <br><br>Some forms of fileless storage activity may indirectly create artifacts in the file system, but in central and otherwise difficult to inspect formats such as the WMI (e.g., `%SystemRoot%\System32\Wbem\Repository`) or Registry (e.g., `%SystemRoot%\System32\Config`) physical files.[^3]
