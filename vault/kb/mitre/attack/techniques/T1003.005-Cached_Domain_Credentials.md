## Description

Adversaries may attempt to access cached domain credentials used to allow authentication to occur in the event a domain controller is unavailable.[^2] <br><br>On Windows Vista and newer, the hash format is DCC2 (Domain Cached Credentials version 2) hash, also known as MS-Cache v2 hash.[^1]  The number of default cached credentials varies and can be altered per system. This hash does not allow pass-the-hash style attacks, and instead requires [[kb/mitre/attack/techniques/T1110.002-Password_Cracking\|Password Cracking]] to recover the plaintext password.[^8] <br><br>On Linux systems, Active Directory credentials can be accessed through caches maintained by software like System Security Services Daemon (SSSD) or Quest Authentication Services (formerly VAS). Cached credential hashes are typically located at `/var/lib/sss/db/cache.[domain].ldb` for SSSD or `/var/opt/quest/vas/authcache/vas_auth.vdb` for Quest. Adversaries can use utilities, such as `tdbdump`, on these database files to dump the cached hashes and use [[kb/mitre/attack/techniques/T1110.002-Password_Cracking\|Password Cracking]] to obtain the plaintext password.[^5]  <br><br>With SYSTEM or sudo access, the tools/utilities such as [[kb/mitre/attack/software/S0002-Mimikatz\|Mimikatz]], [[kb/mitre/attack/software/S0075-Reg\|Reg]], and secretsdump.py for Windows or Linikatz for Linux can be used to extract the cached credentials.[^5] <br><br>Note: Cached credentials for Windows Vista are derived using PBKDF2.[^1]

## Mitigations
| ID | Name | Description |
| --- | --- | --- |
| [[kb/mitre/attack/mitigations/M1015-Active_Directory_Configuration\|M1015]] | Active Directory Configuration | Consider adding users to the "Protected Users" Active Directory security group. This can help limit the caching of users' plaintext credentials.[^6]  |
| [[kb/mitre/attack/mitigations/M1017-User_Training\|M1017]] | User Training | Limit credential overlap across accounts and systems by training users and administrators not to use the same password for multiple accounts. |
| [[kb/mitre/attack/mitigations/M1026-Privileged_Account_Management\|M1026]] | Privileged Account Management | Do not put user or admin domain accounts in the local administrator groups across systems unless they are tightly controlled, as this is often equivalent to having a local administrator account with the same password on all systems. Follow best practices for design and administration of an enterprise network to limit privileged account use across administrative tiers. |
| [[kb/mitre/attack/mitigations/M1027-Password_Policies\|M1027]] | Password Policies | Ensure that local administrator accounts have complex, unique passwords across all systems on the network. |
| [[kb/mitre/attack/mitigations/M1028-Operating_System_Configuration\|M1028]] | Operating System Configuration | Consider limiting the number of cached credentials (HKLM\SOFTWARE\Microsoft\Windows NT\Current Version\Winlogon\cachedlogonscountvalue)[^10]  |
