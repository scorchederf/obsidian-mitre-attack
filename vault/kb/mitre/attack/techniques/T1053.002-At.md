## Description

Adversaries may abuse the [[kb/mitre/attack/software/S0110-at\|at]] utility to perform task scheduling for initial or recurring execution of malicious code. The [[kb/mitre/attack/software/S0110-at\|at]] utility exists as an executable within Windows, Linux, and macOS for scheduling tasks at a specified time and date. Although deprecated in favor of [[kb/mitre/attack/techniques/T1053.005-Scheduled_Task\|Scheduled Task]]'s [[kb/mitre/attack/software/S0111-schtasks\|schtasks]] in Windows environments, using [[kb/mitre/attack/software/S0110-at\|at]] requires that the Task Scheduler service be running, and the user to be logged on as a member of the local Administrators group. In addition to explicitly running the `at` command, adversaries may also schedule a task with [[kb/mitre/attack/software/S0110-at\|at]] by directly leveraging the [[kb/mitre/attack/techniques/T1047-Windows_Management_Instrumentation\|Windows Management Instrumentation]] `Win32_ScheduledJob` WMI class.[^8] <br><br>On Linux and macOS, [[kb/mitre/attack/software/S0110-at\|at]] may be invoked by the superuser as well as any users added to the `at.allow` file. If the `at.allow` file does not exist, the `at.deny` file is checked. Every username not listed in `at.deny` is allowed to invoke [[kb/mitre/attack/software/S0110-at\|at]]. If the `at.deny` exists and is empty, global use of [[kb/mitre/attack/software/S0110-at\|at]] is permitted. If neither file exists (which is often the baseline) only the superuser is allowed to use [[kb/mitre/attack/software/S0110-at\|at]].[^6] <br><br>Adversaries may use [[kb/mitre/attack/software/S0110-at\|at]] to execute programs at system startup or on a scheduled basis for [[kb/mitre/attack/tactics/TA0003-Persistence\|Persistence]]. [[kb/mitre/attack/software/S0110-at\|at]] can also be abused to conduct remote [[kb/mitre/attack/tactics/TA0002-Execution\|Execution]] as part of [[kb/mitre/attack/tactics/TA0008-Lateral_Movement\|Lateral Movement]] and/or to run a process under the context of a specified account (such as SYSTEM).<br><br>In Linux environments, adversaries may also abuse [[kb/mitre/attack/software/S0110-at\|at]] to break out of restricted environments by using a task to spawn an interactive system shell or to run system commands. Similarly, [[kb/mitre/attack/software/S0110-at\|at]] may also be used for [[kb/mitre/attack/tactics/TA0004-Privilege_Escalation\|Privilege Escalation]] if the binary is allowed to run as superuser via `sudo`.[^5]

## Mitigations
| ID | Name | Description |
| --- | --- | --- |
| [[kb/mitre/attack/mitigations/M1018-User_Account_Management\|M1018]] | User Account Management | Limit privileges of user accounts and remediate Privilege Escalation vectors so only authorized administrators can create scheduled tasks on remote systems. In Linux environments, users account-level access to `[[kb/mitre/attack/software/S0110-at\|at]]` can be managed using `at.allow` and `at.deny` files. Users listed in the at.allow are enabled to schedule actions using at, whereas users listed in at.deny file disabled from the utility. |
| [[kb/mitre/attack/mitigations/M1026-Privileged_Account_Management\|M1026]] | Privileged Account Management | Configure the Increase Scheduling Priority option to only allow the Administrators group the rights to schedule a priority process. This can be configured through GPO: Computer Configuration > [Policies] > Windows Settings > Security Settings > Local Policies > User Rights Assignment: Increase scheduling priority. [^13]  |
| [[kb/mitre/attack/mitigations/M1028-Operating_System_Configuration\|M1028]] | Operating System Configuration | Configure settings for scheduled tasks to force tasks to run under the context of the authenticated account instead of allowing them to run as SYSTEM. The associated Registry key is located at `HKLM\SYSTEM\CurrentControlSet\Control\Lsa\SubmitControl`. The setting can be configured through GPO: Computer Configuration > [Policies] > Windows Settings > Security Settings > Local Policies > Security Options: Domain Controller: Allow server operators to schedule tasks, set to disabled. [^12]   |
| [[kb/mitre/attack/mitigations/M1047-Audit\|M1047]] | Audit | Toolkits like the PowerSploit framework contain PowerUp modules that can be used to explore systems for permission weaknesses in scheduled tasks that could be used to escalate privileges. [^2]  Windows operating system also creates a registry key specifically associated with the creation of a scheduled task on the destination host at: Microsoft\Windows NT\CurrentVersion\Schedule\TaskCache\Tree\At1. [^16]  In Linux and macOS environments, scheduled tasks using `[[kb/mitre/attack/software/S0110-at\|at]]` can be audited locally, or through centrally collected logging, using syslog, or auditd events from the host. [^15]  |
