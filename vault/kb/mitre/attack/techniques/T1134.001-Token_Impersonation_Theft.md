## Description

Adversaries may duplicate then impersonate another user's existing token to escalate privileges and bypass access controls. For example, an adversary can duplicate an existing token using `DuplicateToken` or `DuplicateTokenEx`.[^4]  The token can then be used with `ImpersonateLoggedOnUser` to allow the calling thread to impersonate a logged on user's security context, or with `SetThreadToken` to assign the impersonated token to a thread.<br><br>An adversary may perform [[kb/mitre/attack/techniques/T1134.001-Token_Impersonation_Theft\|Token Impersonation/Theft]] when they have a specific, existing process they want to assign the duplicated token to. For example, this may be useful for when the target user has a non-network logon session on the system.<br><br>When an adversary would instead use a duplicated token to create a new process rather than attaching to an existing process, they can additionally [[kb/mitre/attack/techniques/T1134.002-Create_Process_with_Token\|Create Process with Token]] using `CreateProcessWithTokenW` or `CreateProcessAsUserW`. [[kb/mitre/attack/techniques/T1134.001-Token_Impersonation_Theft\|Token Impersonation/Theft]] is also distinct from [[kb/mitre/attack/techniques/T1134.003-Make_and_Impersonate_Token\|Make and Impersonate Token]] in that it refers to duplicating an existing token, rather than creating a new one.

[^3]  Also define who can create a process level token to only the local and network service through GPO: Computer Configuration > [Policies] > Windows Settings > Security Settings > Local Policies > User Rights Assignment: Replace a process level token.[^1] <br><br>Administrators should log in as a standard user but run their tools with administrator privileges using the built-in access token manipulation command `runas`.[^5]  |
